# Choose-the-Right-DB


      
<p>How do you choose a database? Maybe, you assess whether the use case needs a Relational database. Depending on the answer, you pick your favorite SQL or NoSQL datastore, and make it work. It is a prudent tactic: a known devil is better than an unknown angel.</p>
<p>Picking the right datastore can simplify your application. A wrong choice can add friction. This article will help you expand your list of known devils with an in-depth overview of various datastores. It covers the following:</p>
<ul>
<li><strong>Database parts</strong> that define a datastore’s characteristics.</li>
<li><strong>Datastores</strong> categorized by <strong>data types</strong>: deep dive into databases for unstructured, structured (SQL/tabular), and semi-structured (NoSQL) data.</li>
<li><strong>When</strong> to use <strong>NoSQL</strong> vs. <strong>SQL</strong> database.</li>
<li><strong>Differences</strong> between <strong>SQL</strong> and <strong>NoSQL</strong> databases.</li>
<li><strong>Datastores</strong> specialized for various <strong>NoSQL use cases</strong>.</li>
<li><strong>Decision Tree Cheatsheet</strong> to navigate the landscape of on-prem and on-cloud datastore choices.</li>
</ul>

<p>A high-level understanding of how databases work helps in evaluating alternatives. Databases have 5 components: interface, query processor, metadata, indexes, and storage:</p>
<ol>
<li><strong>Interface Language or API:</strong> Each database defines a language or API to interact with it. It covers definition, manipulation, query, and control of data and transactions.</li>
<li><strong>Query Processor:</strong> The “CPU” of the database. Its job is to process incoming requests, perform needed actions, and return results.</li>
<li><strong>Storage:</strong> The disk or memory where the data is stored.</li>
<li><strong>Indexes:</strong> Data structures to quickly locate the queried data in the storage.</li>
<li><strong>Metadata:</strong> Meta-information of data, storage. and indexes (e.g., catalog, schema, size).</li>
</ol>
<p>The <strong>Query Processor</strong> performs the following steps for each incoming request:</p>
<ol>
<li>Parses the request and validates it against the metadata.</li>
<li>Creates an efficient execution plan that exploits the indexes.</li>
<li>Reads or updates the storage.</li>
<li>Updates metadata and indexes.</li>
<li>Computes and returns results.</li>
</ol>
<p>To determine a datastore matches your application needs, you need carefully examine:</p>
<ul>
<li><strong>Operations</strong> supported by the interface. If the computations you require are in-built, you will need to write less code.</li>
<li>Available <strong>indexes</strong>. It will determine how fast your queries run.</li>
</ul>

<p>The file system is the simplest and oldest datastore. We use it every day to store all kinds of data. Blob Storage is a hyper-scale distributed version of the filesystem. It is used to store <em>unstructured data</em>.</p>
<p>Blob’s <a href="https://en.wikipedia.org/wiki/Backronym">backronym</a> is Binary Large OBjects. You can store any kind of data. Therefore blob datastore has no role in interpreting the data:</p>
<ul>
<li>Blob supports CRUD (create, read, update, delete) <strong>operations</strong> at the file level.</li>
<li>The directory or file <em>path</em> is the <strong>index</strong>.</li>
</ul>
<p>So you can quickly locate and read the file. But locating something within a file requires a sequential scan. Documents, images, audio, and video files are stored in blobs.</p>

<p>SQL databases are suitable for storing <em>structured data</em>. Data is stored in a table. Each record (<em>row</em>) has the same number of attributes (<em>columns</em>) of the same type.</p>
<p>There are two kinds of applications:</p>
<ul>
<li>Online <strong>Transaction</strong> Processing (<strong>OLTP</strong>): Capture, store, and process data from transactions in real-time.</li>
<li>Online <strong>Analytical</strong> Processing (<strong>OLAP</strong>): analyze aggregated historical data from OLTP applications.</li>
</ul>
<p>OLTP applications need datastores that support <em>low latency</em> reads and writes of <em>individual</em> records. OLAP applications need datastores that support <em>high throughput</em> reads on a large number of (<em>read-only</em>) records.</p>

<p>Relation Database Management Systems (RDBMS) are one of the earliest datastores. RDBMS are optimized for OLTP workloads requiring fast reading and updating a large number of rows. That’s why RDBMSs are row-oriented databases.</p>
<p>The data is organized in tables. Tables are <a href="https://en.wikipedia.org/wiki/Database_normalization">normalized</a> for reduced data redundancy and better data integrity.</p>
<p>Tables may have primary and foreign keys:</p>
<ul>
<li>
<p><a href="https://en.wikipedia.org/wiki/Primary_key"><strong>Primary Key</strong></a> is a minimal set of attributes (columns) that uniquely identifies a record (row) in a table.</p>
</li>
<li>
<p><a href="https://en.wikipedia.org/wiki/Foreign_key"><strong>Foreign Key</strong></a> establishes relationships between tables. It is a set of attributes in a table that refers to the primary key of another table.</p>
</li>
</ul>
<p>Query and transactions are coded using Standard Query Language (SQL).</p>
<p>Relational databases are optimized for transaction operations. Transactions often update multiple records in multiple tables. Indexes are optimized for frequent low-latency writes of <a href="https://en.wikipedia.org/wiki/ACID">ACID Transactions</a>:</p>
<ul>
<li>
<p><strong>Atomicity:</strong> Any transaction that updates multiple rows is treated as a single <em>unit</em>. A successful transaction performs <em>all</em> updates. A failed transaction performs <em>none</em> of the updates, i.e., the database is left unchanged.</p>
</li>
<li>
<p><strong>Consistency:</strong> Every transaction brings the database from one valid state to another. It guarantees to maintain all database invariants and constraints.</p>
</li>
<li>
<p><strong>Isolation:</strong> Concurrent execution of multiple transactions leaves the database in the same state as if the transactions were executed sequentially.</p>
</li>
<li>
<p><strong>Durability:</strong> Committed transactions are permanent, and survive even a system crash.</p>
</li>
</ul>

<ul>
<li>Cloud Agnostic: Oracle, Microsoft SQL Server, IBM DB2, <a href="https://www.postgresql.org/">PostgreSQL</a>, and <a href="https://www.mysql.com/">MySQL</a></li>
<li>AWS: Hosted PostgreSQL and MySQL in <a href="https://aws.amazon.com/rds/">Relational Database Service (RDS)</a></li>
<li>Microsoft Azure: Hosted SQL Server as <a href="https://azure.microsoft.com/en-in/products/azure-sql/database/">Azure SQL Database</a></li>
<li>Google Cloud: Hosted PostgreSQL and MySQL in <a href="https://cloud.google.com/sql/">Cloud SQL</a>, and also horizontally scaling <a href="https://cloud.google.com/spanner">Cloud Spanner</a></li>
</ul>

<p>While transactions are on rows (records), analytics properties are computed on columns (attributes). OLAP applications need an optimized column-read operation on a table. Columnar databases are designed for high-throughput of column aggregations. That’s why Columnar DBs are row-oriented databases.</p>
<p>One way to achieve it is by adding column-oriented indexes to Relational databases. For example:</p>
<ul>

<p>However, the primary RDBMS operation is low-latency high-frequency ACID transactions. That does not scale to the Big Data scale common in analytics applications.</p>

<p>A key-value store is a <a href="https://en.wikipedia.org/wiki/Associative_array">dictionary or hash table</a> database. It is designed for CRUD operations with a unique key for each record:</p>
<ul>
<li>Create(key, value): Add a key-value pair to the datastore</li>
<li>Read(key): Lookup the value associated with the key</li>
<li>Update(key, value): Change the existing value for the key</li>
<li>Delete(key): Delete the (key, value) record from the datastore</li>
</ul>
<p>The values do not have a fixed schema and can be anything from primitive values to compound structures. Key-value stores are highly partitionable (thus scale horizontally). <a href="https://redis.io/">Redis</a> is a popular key-value store.</p>

<p>A wide-column store has tables, rows, and columns. But the names of the columns and their types may be different for each row in the same table. Logically, It is a versioned sparse matrix with multi-dimensional mapping (row-value, column-value, timestamp). It is like a two-dimensional key-value store, with each cell value versioned with a timestamp.</p>
<p>Wide-column datastores are highly partitionable. It has a notion of column families that are stored together. The logical coordinates of a cell are: (Row Key, Column Name, Version). The physical lookup is as following: Region Dictionary ⇒ Column Family Directory ⇒ Row Key ⇒ Column Family Name ⇒ Column Qualifier ⇒ Version. So, wide-column stores are actually row-oriented databases.</p>
<p><a href="https://hbase.apache.org/">Apache HBase</a> was the first open-source wide-column datastore. Check out <a href="https://www.slideshare.net/larsgeorge/hbase-in-practice">HBase in Practice</a>, for core concepts of wide-column datastores.</p>

<p>Non-relational NoSQL datastores gained popularity for two reasons:</p>
<ul>
<li>
<p>RDBMS did not scale horizontally for Big Data</p>
</li>
<li>
<p>Not all data fits into strict RDBMS schema</p>
</li>
</ul>
<p>NoSQL datastores offer horizontal scale at various CAP Theorem tradeoffs. As per <a href="https://en.wikipedia.org/wiki/CAP_theorem">CAP Theorem</a>, a distributed datastore can give at most 2 of the following 3 guarantees:</p>
<ul>
<li>
<p><strong>Consistency:</strong> Every read receives the most recent write or an error.</p>
</li>
<li>
<p><strong>Availability:</strong> Every request gets a (non-error) response, regardless of the individual states of the nodes.</p>
</li>
<li>
<p><strong>Partition tolerance:</strong> The cluster does not fail despite an arbitrary number of messages being dropped (or delayed) by the network between nodes.</p>
</li>
</ul>
<p>Note that the consistency definitions in CAP Theorem and ACID Transactions are different. ACID consistency is about data integrity (data is consistent w.r.t. relations and constraints after every transaction). CAP is about the state of all nodes being consistent with each other at any given time.</p>
<p>Only a few NoSQL datastores are ACID-complaint. Most NoSQL datastore support <a href="https://dl.acm.org/doi/10.1145/1394127.1394128">BASE model</a>:</p>
<ul>
<li>
<p><strong>Basically Available:</strong> Data is replicated on many storage systems and is available most of the time.</p>
</li>
<li>
<p><strong>Soft-state:</strong> Replicas are not consistent all the time; so the state may only be partially correct as it may not yet have converged.</p>
</li>
<li>
<p><strong>Eventually consistent:</strong> Data will become consistent at some point in the future, but no guarantee when.</p>
</li>
</ul>

<p>Differences between RDBMS and NoSQL databases stem from their choices for:</p>
<ul>
<li>
<p><strong>Data Model:</strong> RDBMS databases are used for normalized structured (tabular) data strictly adhering to a relational schema. NoSQL datastores are used for non-relational data, e.g. key-value, document tree, graph.</p>
</li>

</li>
</ul>

<p>RDBMS are designed for fast transactions updating multiple rows across tables with complex integrity constraints. SQL queries are expressive and declarative. You can focus on <em><strong>what</strong></em> a transaction should accomplish. RDBMS will figure out <em><strong>how</strong></em> to do it. It will optimize your query using relational algebra and find the best execution plan.</p>
<p>NoSQL datastores are designed for efficiently handling a lot more data than RDBMS. There are no relational constraints on the data, and it does not need to be even tabular. NoSQL offers performance at a higher scale by typically giving up strong consistency. Data access is mostly through REST APIs. NoSQL query languages (such as GraphQL) are not yet as mature as SQL in design and optimizations. So you need to take care of both <em>what</em> and <em>how</em> to do it efficiently.</p>
<p><strong>RDBMS scale vertically.</strong> You need to upgrade hardware (more powerful CPU, higher storage capacity) to handle the increasing load.</p>
<p><strong>NoSQL datastores scale horizontally.</strong> NoSQL is better at handling partitioned data, so you can scale by adding more machines.</p>
